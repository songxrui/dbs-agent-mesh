# EXAMPLE_01: dbs-agent-mesh 实战 — 知识付费项目全流程诊断

## 输入
```
用户: "我做了一个知识付费产品，定价199，卖了2个月才30单，感觉卡住了。帮我从头到尾分析一下。"
```

## 流水线编排

### 选择的流水线: A（商业模式深度诊断）

```
Phase 1: dbs-diagnosis → "商业模式消解"
  输出: 真正的问题不是"定价太低"而是"用户获取和留存"

Phase 2: dbs-deconstruct → "拆解'知识付费'在这个上下文里的含义"
  输出: 用户买的是'结果'不是'知识'——拆解到原子概念

Phase 3: dbs-goal → "审计目标：30单/2月 vs 期望目标"
  输出: 目标从"卖更多"审计为"月均50单，复购率>20%"

Phase 4: dbs-benchmark → "对标分析：同类知识付费产品的定价和转化"
  输出: 对标 3 个竞品 → 定价在合理区间，问题在流量端

Phase 5: dbs-decision → "矩阵决策：涨价 vs 降价 vs 捆绑销售"
  输出: 推荐捆绑销售（产品A+社群），涨价到299

Phase 6: dbs-action → "执行计划：14天实验"
  输出: Day1-3 改销售页, Day4-7 社群冷启动, Day8-14 测转化率

Phase 7: dbs-save → "存档全流程结论"
  输出: 7 步诊断结论已存档，随时 dbs-restore
```

## 每个 Phase 的输入输出传递
| Phase | 输入（来自上一Phase） | 输出（传给下一Phase） |
|-------|---------------------|---------------------|
| diagnosis | 用户原始描述 | 重新定义的问题 |
| deconstruct | 重新定义的问题 | 原子化概念 |
| goal | 原子化概念 + 问题 | 可量化的目标 |
| benchmark | 量化目标 | 对标数据 |
| decision | 对标数据 + 目标 | 决策矩阵 |
| action | 决策结论 | 执行计划 |
| save | 全部输出 | 存档 |

## 验证
- ✅ 7 个 phase 串行执行，前输出后输入
- ✅ 每 phase 明确调用哪个 skill
- ✅ 用户可随时查看中间结果
- ✅ 最终存档，可恢复
